Project Documentation

Onboarding,
as we knew it,
is dead.

Post‑AI, users don't need AI explained to them — they need an instant AHA moment. This project reimagines onboarding as progressive intelligence that learns from what the user does during creation, not from a survey after it, and turns the second session into proof the tool was paying attention.

01

The problem

A mental‑model shift, not a UX inefficiency.

Traditional onboarding ran on one assumption: users need to understand the tool before they can use it. That held when interfaces were complex, capabilities were thin, and people had time to learn. The AI era broke all three conditions at once.

  • Users now expect intelligence, not education. "Show me how" became "just do it for me."
  • Tool saturation collapsed evaluation time from minutes to seconds — people bounce before a tour can demonstrate value.
  • A generic experience now reads as broken: "If Netflix can predict what I want to watch, why can't this tool predict what I want to create?"

The reframe

Old problem: "How do we teach users what AI can do?"
New problem: "How do we get users to their AHA moment before they bounce?"

02

Why we started

The hypothesis, and where it came from.

The seed was applying the Claude interaction pattern to product onboarding — understand intent, ask smart contextual questions, deliver exactly what's needed, remember for next time. Not copying a UI; applying the intelligence pattern.

The hypothesis

If we gather context during creation rather than before or after, we can:

  1. Make the first experience feel intelligent rather than generic.
  2. Create natural pathways to advanced features (visual diffs, troubleshooting, agentflows).
  3. Build progressive intelligence that gets smarter every visit.
  4. Collapse the old "onboard → discover features → become a power user" funnel into one continuous loop.
Ruben's POV — the framing that changed the project

"Users don't need AI explained anymore. They need instant AHA moments." Once that landed, this stopped being "how do we make onboarding smarter" and became "how do we make onboarding unnecessary by making creation intelligent."

This is positioned as innovation, not iteration: most teams are still optimising the old paradigm — making tutorials more interactive, feature discovery more gamified. The bet here is that the entire category of user education went obsolete when expectations shifted to tool intelligence.

03

Baseline reality check

2026‑05‑04 — re‑anchoring the brief against data that actually exists.

The original brief carried three baseline numbers (60% completion, 8–12 min to first success, 15% feature discovery). On inspection, two of the three don't trace to any current Mixpanel or Statsig source. Treating them as known was overconfident. Here's what the data actually says, anchored to events that exist.

What's actually measured today

From the Cold Start Lookback (~10d post the Apr‑7 redesign): cold‑start action rate is 84% (up from ~66%), method mix is roughly half AI / half paste, blank‑canvas collapsed from ~25% → 8%, total creation volume +~20%. But Trial Started stayed flat — the activation lift didn't propagate downstream in the window.

⚑ What this means for the project

The Cold‑Start redesign already shipped much of what this brief originally described. The risk is that Progressive Onboarding gets read as a re‑do of work already in production. The honest unsolved problem is narrower and sharper: cold‑start lift didn't move trials, and guided‑tour lift didn't move trials. So the success criterion is restated against trial / paid conversion, not session completion.

Open instrumentation needs

  • Define the "success" event explicitly (first accepted AI op / first export / first edit‑after‑render).
  • Build the time‑to‑success funnel — no saved Mixpanel report exists yet.
  • Give "feature discovery" a behavioural definition (used 2+ AI surfaces in session 1?) rather than passive tour‑step events (52% dismiss those).
  • Add a second‑visit retention query (D1/D7 return, onboarded vs not) — the whole "progressive learning" thesis leans on it.

How this lands in the design

  • The checklist's AI item defaults to the highest causal converter — Node‑level AI today. Voice is included but flagged "needs more usage to validate."
  • Sequencing matters: chaining node vibe diagramming → AI chat converts better than the reverse. Surface node AI first, chat as the natural follow‑up — don't randomise the order.
  • Teammate invite is a returning‑user ask, not a first‑run checklist step. The 18× lift validates the lever, but it's correlation‑heavy — so it lives in a persistent "Join Organization" banner aimed at returning users rather than the Get‑Started checklist (see §09).
  • More‑diagrams and personalization are deliberately out of scope — editor team owns the former; the latter is parked until its foundation is ready.
05

The system & thesis

"Context during creation" — three layers of intelligence.

Instead of "What do you want to make?" → "Tell us about yourself" → "Here are some features," the flow is: "Let's make something together" → real‑time intelligence → contextual enhancement → progressive learning. The same arc Claude runs: analyse intent → ask smart questions → deliver exactly what's needed → remember for next time.

Three‑level tool integration

Tool detection has to feel valuable, not creepy. Each level earns trust for the next:

  1. Question enhancement — tool‑specific follow‑ups ("GitHub OAuth scope?" not "Auth type?"). Proves intelligence.
  2. Output modification — tool‑specific diagram elements. Proves value.
  3. Export optimization — formats matching the user's real workflow. Proves workflow understanding.
Architecture decision — 2026‑06‑05

Layer onto the Unified Editor; don't rebuild. The editor already ships OnboardingChecklist.svelte and DiagramStarter.svelte. Onboarding hooks into signals the app already emits (nodes, selectedIds, chat.appliedAt, chat.voiceMode, activeTab) via an observational onboarding.svelte.ts store — no parallel diagram model. The onboarding stopped being a thing to build from scratch and became a thing to activate and extend.

06

Feature adoption pipeline

Why onboarding is the foundation for everything advanced.

Traditional onboarding leaves a gap: users finish the tour, make a simple diagram, and never discover the advanced surfaces. Intelligent onboarding is meant to be a bridge — each stage sets the expectation that makes the next feel natural rather than foreign.

Returning‑user memory, made concrete

The returning state replaces the first‑run starter with "Let's pick up where you left off," mode‑aware across the three editor modes, and — crucially — driven by what the user actually did in session 1, not a fixed template per mode:

  • Build — memory‑derived continuations ("Expand your <last topic>") + Create agent.
  • Freeform — last‑used shapes as one‑click tiles (real glyphs; click drops onto canvas).
  • Code — source‑import entries; first tile reflects the latest GitHub branch pulled.

When there's no single dominant behaviour, the suggestion cards blend across modes. This is the mechanism that makes "progressive learning" concrete rather than aspirational.

08

Key decisions

What we chose, what we rejected, and why.

Decision · real‑time analysis over post‑creation surveys

Rejected: a questionnaire after the first diagram. Chose: analysis during creation with contextual questioning. Why: more authentic user data, immediate value, natural timing — a survey creates a homework feeling and low completion.

Decision · progressive learning over static templates

Considered: smart templates by role/industry. Chose: dynamic, behaviour‑driven templates per individual. Why: creates a switching cost, demonstrates real intelligence, and scales better than a fixed matrix.

Decision · tool integration over generic features

Chose: tool‑specific integration at three levels (questions → elements → exports) over generic export options. Why: it proves workflow understanding, which is what drives adoption of the advanced surfaces.

Decision · checklist items are guided tasks, not jumps

Each item is collapsed → Skip/Start → "in progress" + a sub‑step micro‑checklist (e.g. node‑AI task: Select → Describe → Try versions → Apply → Done). The personalize card is first‑session only and only during loading — confirming "context during creation, never nags."

Decision · localStorage as the persistence layer

Correct here because the prototype is a real local repo — unlike the earlier standalone‑artifact constraint. Sign‑out is a full reset (wipes onboarding progress, cross‑session memory, and every saved diagram) so the next user is a true first‑timer.

Why it landed here

Ruben's call
Share over Invite at step 4

Ruben's own call: for the MVP, Share is the better conversion driver. Share and Invite overlapped on collaboration, so the checklist keeps the lighter, higher‑intent action and lets the heavier ask live elsewhere.

Design framesBuild / Eng
Node‑level & Voice AI deferred — one generic "Edit with AI"

Rather than split the AI step across methods, the MVP uses a single generic "Edit with AI" (chat). Deferring Node‑level and Voice keeps a clean engagement signal — one AI action to measure, not four — without throwing away the richer methods (they stay in the editor, just not as checklist gates).

Data — DominicRuben's call

The invite ask wasn't dropped — it relocated to a persistent "Join Organization" banner on the dashboard and project pages. That's the right home for a heavier, retention‑driving, correlation‑heavy ask aimed at returning users: inviters convert ~8.89% vs 0.49% (18×) and retain ~80% vs 73%, but they're usually already activated — so the lever belongs in front of returning users, not as a first‑run checklist step.

10

What shipped & what's left

Built state as of 2026‑06‑18 — live on Vercel and a GitHub Pages static SPA.

Working in the prototype

  • Personalize card — first‑session only, visible only during the loading screen; each "Yes" re‑enters loading so the next question appears. Confirms "context during creation, never nags."
  • Shapes panel — fully wired, scrolls internally; AWS shapes render as real outline icons with service labels (S3, Lambda, EC2, DynamoDB, SNS, SQS, CloudFront, CloudWatch, IAM). Every shape added is remembered.
  • AI bar "+" menu — attach a file · mention a previous diagram · git repo branch · call a Skill (UI placeholders).
  • Returning user, per tab — now in all three modes: Build continuation cards, Freeform connect/auto‑arrange tiles, Code diagram‑from‑repo entries, each with its own leading icon. Cards seed the AI bar with a prompt + an @‑mention pill pointing at your last diagram.
  • Live generation skeleton — the first diagram assembles node‑by‑node from partial output (flowchart TD/LR + sequence) behind a pink↔purple loading pill, instead of a flat "generating…" box. Sandbox at /loading.
  • AI‑added highlight — a new node lights its own outline in the brand gradient and pulses, while the canvas eases a gentle push‑in to frame the change then back out. Manual zoom stays 1:1; reduced‑motion respected.
  • Dashboard ⇄ editor round‑trip — auto‑save on exit, snapshot cards, auto‑naming from first labelled node, no duplicates, and a genuine first‑run reset on sign‑out (clears both onboarding and session‑memory keys).

Known placeholders / next steps

  • AI "+" menu actions are UI only; GitHub "last pulled branch" is an example string.
  • Saved‑diagram thumbnails use a placeholder image (no live SVG/PNG snapshot yet).
  • Changing an existing node to an AWS kind via the properties panel still makes a rectangle; only the toolbar add‑flow creates AWS icon nodes.
  • Two pieces still parked with no design frame: a conversation‑ordered post‑output quick‑action box (AI modes ordered by conversion), and a quiet tool‑detection sidebar nudge (calm hint, not the old popover that hit 52% dismiss). Decide: design in, or drop.
Ruben's POV — what this project is really about

"The thing I care about isn't the checklist — it's that the second session proves the tool was paying attention. The returning experience has to be driven by what the user actually did in their first session, not a fixed template per mode. Post‑AI, people don't want AI explained to them — they want the tool to already understand them. The dashboard round‑trip is where that promise either lands or doesn't."
